# Handoff Guide — Reference

## Choosing the handoff path

```
Is there a GitHub remote?
  ├── Yes → Is `gh` CLI authenticated?
  │     ├── Yes → Does user prefer GH issue?
  │     │     ├── Yes → Path A (GitHub issue)
  │     │     └── No  → Path B (local docs)
  │     └── No  → Path B (local docs, suggest `gh auth login`)
  └── No  → Path B (local docs)
```

Always ask the user which path they prefer rather than silently choosing.

## Path A — GitHub issue

### Prerequisites check

Run these before attempting issue creation:

```bash
# 1. Verify we're in a git repo
git rev-parse --is-inside-work-tree 2>/dev/null || echo "NOT A GIT REPO"

# 2. Verify GitHub remote exists
git remote get-url origin 2>/dev/null || echo "NO REMOTE"

# 3. Verify gh CLI is available and authenticated
gh auth status 2>/dev/null || echo "GH NOT AUTHENTICATED"
```

If any check fails, inform the user and fall back to Path B.

### Composing the issue body

Build a temporary file that stitches the deliverables together:

```bash
cat > /tmp/issue-body.md << 'ISSUE_EOF'
## Specification

$(cat docs/<feature-slug>/SPEC.md)

---

## Implementation Plan

$(cat docs/<feature-slug>/PLAN.md)

---

## Execution notes

- Generated by `/new-plan` — approved by user on [date]
- Execute units with `/new-task <issue-number>` or
  `/new-task docs/<feature-slug>/`
- Each unit lists tests first — write those before production code
- Trace every PR back to requirement IDs (FR-x, AC-x) in this issue
- `/new-task` will add a verification report when all units are complete

ISSUE_EOF
```

### Label strategy

Apply labels that make the issue filterable:

| Label                      | Purpose                         |
| -------------------------- | ------------------------------- |
| `spec`                     | Contains a formal specification |
| `plan`                     | Contains an implementation plan |
| `ready-for-implementation` | Approved and ready for pickup   |
| `claud`                    | Written by Claude               |

If labels don't exist yet, create them:
```bash
gh label create spec --description "Contains a formal specification" --color 0075ca 2>/dev/null
gh label create plan --description "Contains an implementation plan" --color 006b75 2>/dev/null
gh label create ready-for-implementation --description "Approved for pickup" --color 0e8a16 2>/dev/null
gh label create calude --description "Written by Claude" --color 0e8a75 2>/dev/null
```

### After creation

1. Report the issue URL and number to the user
2. Remind the user they can run `/new-task <issue-number>` when ready
3. **STOP. Do NOT begin implementation or invoke `/new-task`.**

## Path B — Local docs directory

### Directory structure

```
docs/<feature-slug>/
├── SPEC.md            # Specification (source of truth)
└── PLAN.md            # Implementation plan with unit breakdown
```

Note: `/new-task` will add VERIFICATION.md to this directory after
execution is complete.

### Feature slug convention

Derive from the feature name:
- Lowercase
- Replace spaces with hyphens
- Remove special characters
- Keep it short but descriptive, 30 char max

Examples:
- "User Authentication System" → `user-auth`
- "CSV Import Pipeline" → `csv-import`
- "Real-time Notifications" → `realtime-notifications`

### File preparation

Add a frontmatter header to each file for context:

```markdown
---
feature: <feature-name>
created: <date>
approved: <date>
status: approved
generated-by: /new-plan
---
```

### Git integration

If in a git repo (even without GitHub remote):

```bash
git add docs/<feature-slug>/
git commit -m "docs(<feature-slug>): add approved spec and plan

Approved by user on <date>.
Contains:
- SPEC.md: X functional requirements, Y acceptance criteria
- PLAN.md: N implementation units
Execute with: /new-task docs/<feature-slug>/"
```

## Consuming the handoff with /new-task

The deliverables are designed for `/new-task` to consume:

### From a GH issue
```
/new-task #<issue-number>
```

### From local docs
```
/new-task docs/<feature-slug>/
```

`/new-task` will:
1. Read SPEC.md and PLAN.md from the source
2. Create a worktree and feature branch
3. Execute units sequentially using TDD
4. Run verification against the spec
5. Update docs
6. Submit via PR (GitHub) or commit (local)
